# XML Generator Tool — Гайд для начинающих

Локальный инструмент для QA-инженеров: генерирует XML по DTD-схеме, заполняет тестовыми данными (Faker / LLM / БД) и проверяет результат.

## Что умеет инструмент

| Возможность | Описание |
|-------------|----------|
| Загрузка схемы | `.dtd`, `.jar` (Java-архив со схемой) или Nexus Maven |
| Три режима сборки | Минимальный, Максимальный, Свой (выбор веток дерева) |
| Четыре стратегии заполнения | Faker, AI/LLM, Гибрид: БД + Faker, Гибрид: БД + AI |
| Валидация по DTD | С переходом к проблемной строке |
| Библиотека эталонов | Синхронизация с Git-репозиторием |
| Сравнение с эталонами | Проверка структурной уникальности + AI-объяснение |
| Многопользовательский режим | У каждого своё пространство: схемы, пресеты, алиасы |
| Поддержка БД | PostgreSQL и Oracle (12.1+ нативно, 11g через Instant Client) |

**Типовой сценарий работы:**

```
Загрузить DTD → Выбрать корневой элемент и режим сборки → Сгенерировать каркас
→ Выбрать стратегию заполнения → Заполнить данными → Проверить по DTD → Экспортировать
```

## Установка

> Выберите один из двух вариантов: **Docker** (проще) или **ручная установка** (нужна для разработки).

### Вариант А — Docker (рекомендуется)

**Требования:** Docker Desktop.

```bash
# 1. Скопировать конфигурацию
copy config\app.json.example config\app.json

# 2. Создать папку для данных
mkdir data

# 3. Собрать и запустить
docker compose up --build -d
```

Откройте **http://localhost:8080** — готово.

> **Важно для Docker:** если PostgreSQL или Ollama стоят на хосте, в алиасах используйте `host.docker.internal` вместо `localhost`.

### Вариант Б — ручная установка

**Требования:** Python 3.10+, Node.js 18+.

```bash
# Backend
cd backend
python -m venv .venv
.venv\Scripts\activate        # Windows
pip install -r requirements-dev.txt

# Frontend
cd ..\frontend
npm install
```

Конфигурация:

```bash
copy config\app.json.example config\app.json
```

## Запуск

### Разработка (два терминала)

```bash
# Терминал 1 — API
cd backend
uvicorn app.main:app --host 0.0.0.0 --port 8080 --reload

# Терминал 2 — UI
cd frontend
npm run dev
```

Откройте **http://localhost:5173**

### Docker

```bash
docker compose up -d               # запуск
docker compose logs -f app         # логи
docker compose down                # остановка
docker compose up --build -d       # пересборка после изменений
```

Откройте **http://localhost:8080**

### Локальный production (без Docker)

```bash
cd frontend && npm run build
cd ..\backend && uvicorn app.main:app --host 0.0.0.0 --port 8080
```

Откройте **http://localhost:8080**

## Первый вход и настройки

### Логин

Приложение работает в **многопользовательском режиме**: каждый входит под своим логином (пароль не нужен). При первом входе система предложит подтвердить создание аккаунта — это защита от опечаток.

У каждого пользователя изолированное пространство: свои DTD-схемы, маппинг-пресеты, сохранённые XML-документы.

### Страница «Настройки»

Здесь настраиваются алиасы для трёх типов подключений:

**Базы данных**

| Поле | Значение |
|------|----------|
| Алиас | Произвольное имя (например, `PROD_PG`) |
| Driver | `postgresql` или `oracle` |
| Host | Адрес сервера (`localhost` или `host.docker.internal` в Docker) |
| Port | `5432` (PG) или `1521` (Oracle) |
| Database | Имя БД или service name (Oracle) |
| User / Password | Учётные данные |

После сохранения нажмите **«Проверить»** — убедитесь, что подключение работает.

> Пароли хранятся на сервере в `data/users/{id}/connections.json`. В UI видны только алиасы.

**LLM**

| Поле | Значение |
|------|----------|
| Алиас | Например, `ollama-local` |
| Base URL | `http://localhost:11434/v1` (Ollama) или адрес OpenAI-совместимого API |
| API Key | Токен (для локального Ollama — произвольная строка) |
| Model | Название модели, например `llama3` или `gpt-4o-mini` |

**Git (эталонная библиотека)**

Нужен, если команда хранит XML-эталоны в Git-репозитории. Задайте личный токен доступа (GitLab: пользователь `oauth2` по умолчанию).

## Загрузка DTD-схемы

### Способы загрузки

На главной странице (Генератор) в левой панели:

| Способ | Когда использовать |
|--------|-------------------|
| Загрузить `.dtd` | Один или несколько DTD-файлов (`.dtd`, `.ent`, `.mod`) |
| Загрузить `.jar` | Java-архив со схемой внутри `META-INF/dtd/` |
| Получить из Nexus | Если DTD публикуется в Maven Nexus (настраивается в `config/app.json`) |

При загрузке нескольких `.dtd`-файлов сразу — они мержатся в одну схему.

### Что происходит после загрузки

- Схема парсится и сохраняется в `data/users/{id}/dtd_schemas/`
- При следующем запуске backend поднимает схему автоматически — повторная загрузка не нужна
- При загрузке новой схемы старая удаляется

## Генерация XML

### Вкладка «Структура»

**1. Выберите корневой элемент** — точка входа в DTD-дерево.

**2. Выберите режим сборки:**

| Режим | Описание |
|-------|----------|
| **Минимальный** | Только обязательные элементы и атрибуты |
| **Максимальный** | Все элементы, включая необязательные и повторяемые (`+`/`*`) |
| **Свой** | Интерактивное дерево — выберите нужные ветки вручную |

**3. В режиме «Свой»** вы увидите всё DTD-дерево с чекбоксами. Ставьте галочки только на нужных ветках. Можно сохранить выбор как **пресет** и использовать повторно.

**4. Нажмите «Сгенерировать XML»** — в правой панели появится XML-каркас с пустыми или placeholder-значениями.

## Заполнение данными

### Вкладка «Данные»

**Шаг 1 — выберите стратегию:**

| Стратегия | Когда использовать |
|-----------|--------------------|
| **Faker** | Быстро, без внешних зависимостей. Данные на русском (ФИО, адреса, ИНН и т.д.) |
| **AI / LLM** | Контекстное заполнение по смыслу поля (нужен настроенный LLM-алиас) |
| **Гибрид: БД + Faker** | Часть полей берётся из SQL-запросов, пустые — добивает Faker |
| **Гибрид: БД + AI** | Часть полей из БД, пустые — добивает LLM |

**Шаг 2 — для гибридных режимов: настройте маппинг**

Маппинг — это связка «SQL-запрос → XML-атрибут»:

1. Нажмите **«Мастер маппинга»**
2. Выберите алиас БД
3. Введите `SELECT`-запрос (только чтение — `UPDATE`/`DELETE` заблокированы)
4. Свяжите колонки результата с XML-атрибутами
5. Сохраните как именованный пресет

> **Фишка:** нажмите «Предложить маппинг» — LLM попробует сам сопоставить колонки с XML-полями по именам.

**Шаг 3 — дополнительные настройки:**

- **Фиксированные значения** — задайте жёсткое значение для конкретного атрибута. Оно не будет перезаписано ни Faker, ни AI.
- **Проверять DTD после заполнения** — автоматически запустит валидацию по завершении.

**Шаг 4 — нажмите «Заполнить данными»**

Ход выполнения виден в прогресс-баре. Долгие операции можно отменить.

## Валидация и редактор

### Редактор (правая панель)

Редактор на базе **Monaco** (тот же, что в VS Code):

| Действие | Как |
|----------|-----|
| Форматировать XML | Alt+Shift+F |
| Валидировать | Кнопка «Проверить по DTD» или автоматически на редактирование (задержка 800 мс) |
| Импортировать XML | Кнопка импорта в тулбаре |
| Скачать XML | Кнопка экспорта |

### Вкладка «Результат»

Показывает:

- **Ошибки валидации** с номерами строк — клик переводит курсор к проблемному месту
- **Предупреждения сборки** (элементы с неоднозначным маппингом)
- **История генераций** — последние 20 результатов из `localStorage`, можно восстановить любой

### Сравнение с эталонами — вкладка «Сравнение»

Если настроена эталонная библиотека:

1. Нажмите «Проверить уникальность» — инструмент сравнит структуру вашего XML со всеми эталонами того же корневого элемента
2. Увидите процент похожести и отличающиеся структурные пути
3. Можно запросить **AI-объяснение** — LLM расскажет, что именно отличается и насколько это значимо с точки зрения тест-покрытия

## Библиотека XML и эталоны

### Вкладка «Библиотека»

Два раздела:

**Эталоны (общая библиотека)**
- Документы из Git-репозитория команды
- Обновить: кнопка «Обновить из Git»
- Структура репозитория: `xml-library/{категория}/{имя}.xml`

**Мои документы (личная библиотека)**
- Сохраняются на сервере в `data/users/{id}/xml_documents/`
- Можно поделиться с коллегой по логину («Поделиться»)
- Кнопка «Открыть» загружает документ в редактор и обновляет кэш для заполнения

### Настройка эталонного Git-репозитория

В `config/app.json`:

```json
{
  "reference_xml": {
    "enabled": true,
    "repo_url": "https://gitlab.company.com/qa/xml-library.git",
    "branch": "main",
    "subdir": "xml-library",
    "cache_dir": "data/reference-xml",
    "push_enabled": true
  }
}
```

Для приватного репозитория каждый пользователь задаёт личный токен на странице Настройки (GitLab — пользователь `oauth2`).

## Фишки и советы

### Пресеты структуры

В режиме «Свой» сохраните выбранное дерево как пресет — не придётся каждый раз выбирать ветки заново. Пресеты привязаны к вашему аккаунту.

### Пресеты маппинга

Настроенный маппинг (SQL → XML) сохраняйте с именем. Пресеты привязаны к алиасу БД, поэтому у разных алиасов — разные наборы пресетов.

### AI-маппинг

Если колонки таблицы называются близко к XML-атрибутам (например, `inn` и атрибут `INN`), нажмите «Предложить маппинг» — LLM сэкономит время на ручной настройке.

### История генераций

Браузер сохраняет до 20 последних результатов (каркас + заполнение). Если что-то случайно перезаписали — восстановите на вкладке «Результат». XML тяжелее 256 КБ хранятся только в рамках сессии.

### Отмена заполнения

Длительное AI-заполнение можно прервать в любой момент кнопкой «Отмена» — уже заполненные поля сохранятся.

### Безопасность SQL

Инструмент принимает только `SELECT`-запросы. PostgreSQL-соединения открываются в режиме `READ ONLY`. `UPDATE`, `DELETE`, `DROP` и множественные команды заблокированы.

### Тёмная тема

Переключатель темы в шапке — выбор сохраняется в браузере.

### Логи и отладка

При проблемах включите подробные логи в `config/app.json`:

```json
{
  "app": {
    "log_level": "DEBUG"
  }
}
```

Логи Docker: `docker compose logs -f app`

## Oracle Instant Client (только для Oracle 11g)

По умолчанию инструмент подключается к Oracle **12.1+** без дополнительных библиотек (thin mode). Если вы работаете с **Oracle 11g** — нужен Instant Client.

**Windows (локально):**

1. Скачайте Oracle Instant Client 19.x (Basic или Light) с сайта Oracle
2. Распакуйте в `C:\Oracle\client19_64\`
3. Убедитесь, что файл `oci.dll` на месте:
   ```powershell
   dir C:\Oracle\client19_64\bin\oci.dll
   ```
4. В `config/app.json` укажите:
   ```json
   {
     "oracle_client_lib_dir": "C:\\Oracle\\client19_64\\bin",
     "oracle_home": "C:\\Oracle\\client19_64",
     "ora_tzfile": null
   }
   ```
5. **Полностью перезапустите backend** (Ctrl+C, затем uvicorn снова)

**Docker:**

1. Скачайте `instantclient-basic-linux.x64-19.31.0.0.0dbru.zip`
2. Положите архив (не распаковывайте) в `docker/oracle/`
3. Пересоберите контейнер:
   ```bash
   docker compose build --no-cache
   docker compose up -d
   ```
4. В `config/app.json`:
   ```json
   {
     "oracle_client_lib_dir": "/opt/oracle/instantclient",
     "oracle_home": "/opt/oracle/instantclient",
     "ora_tzfile": null
   }
   ```

## Решение проблем

| Симптом | Причина | Решение |
|---------|---------|---------|
| `DPY-3010: connections to this database server version are not supported in thin mode` | Oracle 11g без Instant Client | Установите Instant Client, задайте `oracle_client_lib_dir`, полностью перезапустите backend |
| `Oracle thick mode failed to initialize` | Неверный путь или нет `oci.dll` | Проверьте путь и наличие `oci.dll` в `bin/` |
| `ORA-01804: failure to initialize timezone information` | Timezone-файл не найден | Оставьте `"ora_tzfile": null`. Если не помогает — смотрите, какой `.dat` есть в `%ORACLE_HOME%\oracore\zoneinfo\` |
| `Unsupported database driver: oracle` | Старый backend или опечатка в `driver` | Укажите `"driver": "oracle"`, переустановите зависимости |
| `Database alias '…' not found` | Опечатка в алиасе или нет `connections.json` | Проверьте совпадение алиаса в Настройках и в пресете маппинга |
| DTD пропала после перезапуска | Папка `dtd_schemas/` недоступна | Папка `data/` должна монтироваться (Docker) или существовать локально |
| Страница не открывается после `npm run dev` | Backend не запущен | Запустите `uvicorn` в первом терминале, затем `npm run dev` |
| LLM не отвечает | Неверный `base_url` или модель не загружена | Проверьте алиас в Настройках → «Проверить LLM» |
