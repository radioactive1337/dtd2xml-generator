# XML Generator — руководство пользователя

Инструмент для генерации тестовых XML-документов по DTD-схеме, заполнения их данными (Faker, ИИ, БД) и проверки результата. Ниже — как пользоваться уже запущенным приложением: экраны, кнопки и полезные приёмы.

## Как выглядит экран

Экран разделён на две части.

- **Слева** — панель со схемой DTD и пятью вкладками: **Структура**, **Данные**, **Результат**, **Сравнение**, **Библиотека**. Внизу панели — три главные кнопки: **Сгенерировать XML**, **Заполнить данными**, **Проверить по DTD**.
- **Справа** — редактор XML (с подсветкой синтаксиса) и панель кнопок над ним: импорт, форматирование, экспорт, отправка в Git, «Поделиться», сохранение в «Мои документы».

Границу между панелями можно потянуть мышкой, чтобы изменить ширину.

## Типичный сценарий работы

1. Убедиться, что схема DTD загружена — статус виден в шапке слева (имя файла, количество элементов). Если схемы нет или нужна другая — загрузите `.dtd`-файл (или архив со схемой) через блок «Схема DTD».
2. Вкладка **Структура** — выбрать корневой элемент и режим сборки, нажать **«Сгенерировать XML»**. Справа появится каркас документа.
3. Вкладка **Данные** — выбрать стратегию заполнения, при необходимости настроить маппинг из БД, нажать **«Заполнить данными»**.
4. Нажать **«Проверить по DTD»** и посмотреть результат на вкладке **Результат**.
5. При необходимости — донастроить XML вручную в редакторе, экспортировать, сохранить в библиотеку или поделиться с коллегой.

Дальше — подробно про каждый экран.

## Вкладка «Структура»

Здесь задаётся форма (скелет) документа.

- **Корневой элемент** — поле с автодополнением: начните вводить имя элемента, откроется список подходящих вариантов из схемы DTD с кратким описанием (документация из самого DTD).
- **Режим сборки**:

| Режим | Что делает |
| --- | --- |
| Минимальный | Строит только обязательные элементы |
| Максимальный | Добавляет и необязательные элементы, включая повторяющиеся (`+`, `*`) |
| Свой | Открывает дерево схемы, где вручную отмечаются нужные ветки |

- **Число повторов (+ / \*)** — для элементов с квантификатором `+`/`*` можно задать, сколько копий создавать: одно значение «по умолчанию» для всех повторяющихся элементов и отдельные переопределения для конкретных путей (например, 3 контакта, но только 1 адрес).
- В режиме **«Свой»** появляется дерево DTD с чекбоксами: отмечаете нужные элементы, дерево можно искать по имени (поле «Найти элемент»), а набор отметок — сохранить как **пресет**, загрузить сохранённый или прислать коллеге кнопкой «Поделиться».

## Вкладка «Данные»

Здесь задаётся, откуда берутся значения атрибутов и текста.

**Стратегии заполнения:**

| Стратегия | Когда использовать |
| --- | --- |
| Faker | Быстрые случайные тестовые данные без внешних сервисов |
| AI / LLM | ИИ подбирает значения осмысленно, по названию и документации поля |
| Гибрид: БД + Faker | Часть полей — реальные данные из SQL-запроса, остальное — Faker |
| Гибрид: БД + AI | Часть полей — из БД, остальное — генерирует ИИ |

- Если настроено несколько подключений к ИИ — можно выбрать нужный алиас.
- Флажок **«Проверять DTD после заполнения»** — сразу запускает валидацию после заполнения данными, не нужно нажимать отдельную кнопку.
- **«Фиксированные значения полей»** — список «путь → атрибут → значение», который позволяет вручную закрепить конкретное значение для конкретного элемента (например, всегда подставлять один и тот же ИНН). Такие значения имеют наивысший приоритет: **ручное значение → БД → Faker/AI**.
- Для гибридных режимов ниже появляется блок **«Подстановка из БД»** — список SQL-маппингов (какой SQL-запрос заполняет какой элемент). Каждый маппинг настраивается через мастер (кнопка «+ Добавить маппинг» / «Изменить»):
  1. **Элемент** — какой тег заполняем.
  2. **Путь** (необязательно) — если тегов с таким именем в документе несколько, можно указать конкретный (например, `PayDoc.client.contact[0]`), иначе заполнятся все совпадающие.
  3. **SQL** — алиас БД и сам запрос, с кнопкой «Проверить запрос» (показывает реальные колонки и первую строку результата).
  4. **Поля** — сопоставление колонок SQL с атрибутами XML; можно сделать вручную, кнопкой «Добавить все столбцы» или через **«Автосопоставление (AI)»**, которое подбирает соответствия по именам и документации DTD.
  5. **Обзор** — итоговая проверка перед сохранением, с ошибками/предупреждениями, если что-то не так.
- Готовый набор маппингов можно сохранить как **пресет** (по имени БД-алиаса), переиспользовать в других документах, комбинировать несколько пресетов сразу или отправить коллеге кнопкой «Поделиться».

## Кнопки внизу левой панели

- **«Сгенерировать XML»** — строит каркас по настройкам вкладки «Структура».
- **«Заполнить данными»** — запускает выбранную стратегию заполнения. Во время выполнения (особенно для ИИ и БД) показывается прогресс, статус и кнопка **«Отменить»**, если процесс идёт слишком долго.
- **«Проверить по DTD»** — валидирует текущий XML в редакторе на соответствие схеме.

## Вкладка «Результат»

- Цветной статус сверху: всё ок / есть предупреждения / есть ошибки.
- Список ошибок валидации — по клику на строку с ошибкой курсор в редакторе справа сразу переходит к проблемному месту.
- Информация о сборке: сколько узлов создано, есть ли предупреждения (например, о недостижимых или неоднозначных элементах DTD).
- **История генераций** — последние сгенерированные/заполненные версии документа (хранятся в браузере). Каждую можно **восстановить** одним кликом или удалить из списка. История не хранится на сервере и не переживает очистку данных браузера.

## Вкладка «Сравнение»

Позволяет понять, насколько собранный документ похож на «эталонные» примеры для этого корневого элемента.

- Кнопка **«Проверить уникальность»** (находится над редактором XML) сравнивает структуру текущего документа со всеми эталонами того же корневого элемента.
- Результат: «совпадает по структуре» или «уникален» + похожесть на самый близкий эталон в процентах.
- Список уникальных путей (тех элементов/атрибутов, которых нет ни в одном эталоне) — по клику переходит на нужную строку в редакторе; такие элементы дополнительно подсвечиваются оранжевым прямо в редакторе.
- Кнопка **«Объяснить расхождения (ИИ)»** — просит ИИ по-человечески описать, чем документ отличается от эталонов (требует настроенного LLM).

Полезно, когда нужно убедиться, что сгенерированный документ реалистичен и не содержит «выдуманных» ИИ структур, которых не бывает в реальных примерах.

## Вкладка «Библиотека»

Два раздела с переключателем сверху:

- **Эталоны** — общая база примеров XML из общего Git-репозитория, сгруппированная по категориям. Кнопка **«Обновить из Git»** подтягивает свежую версию. Есть поиск по категориям и названиям документов.
- **Мои документы** — личные сохранённые XML-документы. Можно открыть, удалить или поделиться с другим пользователем.

Кнопка **«Открыть»** у любого документа подставляет его в редактор справа.

## Панель кнопок над редактором XML

| Кнопка | Что делает |
| --- | --- |
| Импорт .xml | Загрузить существующий XML-файл в редактор |
| Форматировать | Красиво расставить отступы (также по `Alt+Shift+F`) |
| Очистить | Стереть содержимое редактора |
| Проверить уникальность | Сравнение с эталонами (вкладка «Сравнение») |
| Экспорт .xml | Скачать текущий документ файлом |
| Отправить в Git | Записать документ в Git-репозиторий эталонной библиотеки (путь формируется как `<корневой элемент>/<имя файла>`, при совпадении имени — перезапишет) |
| Поделиться | Отправить документ другому пользователю приложения |
| Сохранить в документы | Сохранить текущий XML в «Мои документы» под своим именем и описанием |

Редактор поддерживает и обычное ручное редактирование — можно править XML прямо руками, не только через генерацию.

## Полезные приёмы («фишки»)

- **Приоритет данных при заполнении**: ручное фиксированное значение → значение из БД → Faker/AI. Это позволяет сгенерировать документ гибридным способом, а один-два «капризных» поля закрепить вручную, не трогая остальную логику.
- **Число повторов можно переопределять точечно**: задайте общее значение «по умолчанию» и отдельно увеличьте/уменьшите повторы только для нужной ветки дерева (например, 5 позиций в заказе, но 1 адрес доставки).
- **Пресеты дерева (режим «Свой»)** и **пресеты SQL-маппинга** экономят время на повторяющихся задачах — сохраняйте один раз и переиспользуйте, а если коллеге нужна такая же настройка — просто отправьте пресет кнопкой «Поделиться», не нужно объяснять на словах.
- **Мастер маппинга «понимает» DTD**: на шаге выбора элемента и полей показывается документация атрибутов прямо из схемы — не нужно открывать DTD-файл отдельно, чтобы понять, что означает атрибут.
- **Автосопоставление полей через ИИ** в мастере маппинга резко ускоряет создание SQL-маппинга: после «Проверить запрос» жмите «Автосопоставление (AI)» — колонки SQL автоматически расставятся по атрибутам XML.
- **Проверка уникальности** — хороший способ отловить ситуацию, когда ИИ или Faker сгенерировали структуру, которой не бывает в реальных документах; сразу видно, какие пути «чужие», и можно перейти прямо к ним в редакторе.
- **История генераций** в браузере — если случайно перезаписали хороший результат новой генерацией, не обязательно всё переделывать: откройте вкладку «Результат» и восстановите нужную версию.
- **Клик по ошибке валидации** — не нужно искать строку глазами, клик по сообщению об ошибке сразу ставит курсор в нужное место редактора.
- **Отмена долгого заполнения**: заполнение через ИИ или большие SQL-выборки может занять время — есть кнопка «Отменить» с прогресс-баром и таймером, чтобы не ждать зря.
- **«Проверять DTD после заполнения»** — включите один раз, и после каждого заполнения данными валидация будет запускаться автоматически, без лишней кнопки.
- **Поделиться можно почти всем**: готовым XML-документом, пресетом дерева, пресетом SQL-маппинга — получателю достаточно быть зарегистрированным пользователем приложения, поиск по имени пользователя работает прямо в диалоге «Поделиться».
- **Библиотека «Эталоны» — это живой репозиторий**: если она давно не обновлялась, нажмите «Обновить из Git» на вкладке «Библиотека», чтобы подтянуть новые примеры от коллег.
- **Не обязательно генерировать всё с нуля**: можно импортировать существующий `.xml`-файл или открыть документ из «Библиотеки» и продолжить работу с ним — заполнение данными и валидация работают и с загруженным документом.
- **Поиск элемента в дереве DTD** (режим «Свой») — если схема большая, не нужно листать вручную: воспользуйтесь полем «Найти элемент», дерево само раскроется и подсветит нужный узел.
